# Database Gap Analysis: SQLite to PostgreSQL Migration

## 1. Executive Summary
The FinSight application currently uses SQLite (locally) and LibSQL/Turso (cloud profile) as its primary database. The goal is to migrate to PostgreSQL. This document outlines the differences, risks, and mitigation strategies for this migration to ensure a smooth transition while preserving data integrity and application functionality.

## 2. Current State (SQLite)
- **Database Engine:** SQLite (local), LibSQL (Turso/cloud).
- **Dialect:** `org.hibernate.community.dialect.SQLiteDialect`
- **Schema Management:** Flyway with standard SQL scripts (e.g., `V1__initial_schema.sql`).
- **Data Types:** Uses basic SQLite types (`INTEGER`, `TEXT`, `REAL`, `DATE`, `DATETIME`).
- **Auto-increment:** `AUTOINCREMENT` keyword in SQL scripts.
- **ORM:** Hibernate/JPA is used extensively with `GenerationType.IDENTITY`.
- **Querying:** Repositories use JPQL; no native SQLite queries (`nativeQuery=true`) were found in the `repository` interfaces.
- **Constraints:** `CHECK` constraints (e.g., `status IN ('PENDING','PROCESSED')`) and `UNIQUE` constraints are used.

## 3. Target State (PostgreSQL)
- **Database Engine:** PostgreSQL (e.g., version 15+).
- **Dialect:** `org.hibernate.dialect.PostgreSQLDialect`
- **Schema Management:** Flyway with PostgreSQL-compatible SQL scripts.
- **Data Types:** Richer, stricter data types (`SERIAL`/`BIGSERIAL`, `VARCHAR`, `NUMERIC`, `TIMESTAMP`, `BOOLEAN`).
- **Auto-increment:** `SERIAL` or `GENERATED ALWAYS AS IDENTITY`.
- **ORM:** Seamless integration with Hibernate; `GenerationType.IDENTITY` remains compatible.

## 4. Gap Analysis

| Feature | SQLite (Current) | PostgreSQL (Target) | Gap / Action Required |
| :--- | :--- | :--- | :--- |
| **Primary Keys** | `INTEGER PRIMARY KEY AUTOINCREMENT` | `SERIAL PRIMARY KEY` or `INTEGER GENERATED BY DEFAULT AS IDENTITY` | Update Flyway migration scripts to replace `AUTOINCREMENT`. |
| **Data Types: Decimals** | `REAL` | `NUMERIC(15,2)` or `DECIMAL(15,2)` | Replace `REAL` with `NUMERIC` for precise financial calculations (e.g., `amount`). |
| **Data Types: Dates/Times** | `DATETIME`, `DATE` | `TIMESTAMP`, `DATE` | Replace `DATETIME` with `TIMESTAMP` in SQL scripts. |
| **Data Types: Booleans** | `INTEGER` (0/1) | `BOOLEAN` | SQLite uses `INTEGER` for booleans (e.g., `reconciled INTEGER NOT NULL DEFAULT 0`). PostgreSQL has a native `BOOLEAN` type. |
| **Enums / Constraints** | `CHECK (type IN (...))` | `CHECK` or `ENUM` types | `CHECK` constraints are fully supported in Postgres, but `VARCHAR` should replace `TEXT` if length limits are desired. |
| **Dialect Configuration** | `SQLiteDialect` | `PostgreSQLDialect` | Update `application.yml` properties and JDBC driver. |
| **Pragmas** | `PRAGMA journal_mode=WAL;` | Not applicable | Remove SQLite-specific PRAGMAs from `V1__initial_schema.sql`. |
| **Data Migration** | Flat files or `.db` file | `pg_dump`/`pg_restore` or ETL tools | Data must be extracted from `.db` and inserted into Postgres via scripts or ETL tools. |

## 5. Risk Assessment

### 5.1 Schema Incompatibility (Medium Risk)
- **Risk:** Flyway scripts written for SQLite will fail on PostgreSQL due to syntax differences (e.g., `AUTOINCREMENT`, `PRAGMA`).
- **Impact:** Application will fail to start if `spring.flyway.enabled=true` with the old scripts.

### 5.2 Boolean Type Mismatch (Low Risk)
- **Risk:** JPA entities might map `Boolean` to `INTEGER` (0/1) natively in SQLite, but expect native `BOOLEAN` in PostgreSQL.
- **Impact:** Potential deserialization or query errors if the schema defines `INTEGER` but JPA expects `BOOLEAN`.

### 5.3 Float / Decimal Precision Loss (High Risk)
- **Risk:** SQLite uses `REAL` (floating-point) for amounts. Financial applications require exact precision (`NUMERIC`/`DECIMAL`).
- **Impact:** Rounding errors in reconciliations if `REAL` is migrated directly to `DOUBLE PRECISION` instead of `NUMERIC`.

### 5.4 Application Configuration Changes (Low Risk)
- **Risk:** Missing driver dependencies or incorrect JDBC URLs.
- **Impact:** Connection failures on deployment.

## 6. Mitigation Strategies

### 6.1 Refactor Flyway Migrations
1. Create a clean base migration for PostgreSQL (e.g., `V1__initial_schema_pg.sql`), replacing `AUTOINCREMENT` with `SERIAL` or `IDENTITY`.
2. Remove all SQLite `PRAGMA` statements.
3. Change `REAL` columns to `NUMERIC(15,2)` to align with JPA `@Column(precision=15, scale=2)`.
4. Change `INTEGER` boolean columns (like `reconciled`, `resolved`) to `BOOLEAN` with `DEFAULT FALSE`.
5. Organize Flyway paths. Use vendor-specific locations: `db/migration/sqlite` and `db/migration/postgresql`.

### 6.2 Application Configuration Updates
1. Add the PostgreSQL JDBC driver (`org.postgresql:postgresql`) to `pom.xml`.
2. Update `application.yml`:
   ```yaml
   spring:
     datasource:
       url: jdbc:postgresql://localhost:5432/finsight
       username: user
       password: password
       driver-class-name: org.postgresql.Driver
     jpa:
       database-platform: org.hibernate.dialect.PostgreSQLDialect
   ```

### 6.3 Data Migration Plan
1. If retaining existing data is necessary, use a tool like `pgloader` (which natively supports SQLite to Postgres migration) or write a custom Python script to export SQLite tables to CSV and `COPY` them into Postgres.
2. If starting fresh, simply drop the old database and let Flyway reconstruct the schema in PostgreSQL.

### 6.4 Testing Strategy
1. Since the application uses standard JPQL and no native queries, the JPA layer should transition smoothly.
2. Run the existing test suite (`mvn test`) using Testcontainers with a PostgreSQL image to verify ORM mapping and repository queries against a real Postgres instance.
